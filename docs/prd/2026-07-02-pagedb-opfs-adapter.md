## Problem Statement

PageDB (`NodeDB-Lab/pagedb`, vendored at `~/src/lodefolio/data/databases/pagedb`) is an encrypted, portable, async-native embedded page store whose README promises "WASM / OPFS first-class" browser storage. The promise is currently unmet: the existing `opfs` backend (`src/vfs/opfs/`) is an async main-thread proxy that serializes every I/O operation over `postMessage` to a pure-JS worker, has **never been executed** (CI only compile-checks it, there is zero wasm test infrastructure in the repo), and contains known correctness and performance defects — per-page-read `Array.from` byte boxing (two copies + O(n) boxing), no per-file handle deduplication despite the engine double-opening `/main.db` on every commit (which must throw `NoModificationAllowedError` under OPFS exclusive locking), a copy+delete `rename` that violates the `Vfs` trait contract, per-tab-only locks (a second tab can silently corrupt the database), and a fire-and-forget close that races re-opens.

Two downstream projects are blocked on a trustworthy browser backend and have already written their requirements against it: **telpher**'s document-editing stack (PRD-035 / `docstore-pagedb.md`: PageDB is the primary DocStore backend for native *and* web, with a hard gate — "do not promote PageDB browser durability until the native + OPFS oracle proves crash/reopen, quota, worker ownership, corruption recovery, and export/migration behavior") and **gitspen**'s `nodedb-lite` browser edge tier (PageDB EventLog SoR backend, built first in the crate sequencing). Both explicitly forbid a COOP/COEP correctness dependency and require a dedicated worker to exclusively own the store.

This project builds the definitive OPFS backend: PageDB running *inside* a dedicated Web Worker calling `FileSystemSyncAccessHandle` synchronously (maximum performance, no message hops in the hot path), the full `Vfs` contract satisfied with honest per-browser durability semantics, cross-tab single-writer safety, an optional cross-thread async bridge, and — the piece that makes it trustworthy — a real browser conformance + durability oracle harness. The backend lands upstream in `NodeDB-Lab/pagedb` as a PR (author is PR-friendly); the harness, benches, and demo live permanently in this repo (`~/code/pres/brow/pagedb-opfs`).

## Core User Flows

### Flow 1: Rust embedder opens a database in the storage worker (Mode 1, default)

1. App (telpher DocStore / gitspen `nodedb-lite`, compiled to `wasm32-unknown-unknown`) spawns the storage worker via the JS bootstrap (worker URL + capability probe).
2. Inside the worker, the embedder constructs `OpfsVfs` and calls `Db::open(OpenOptions { vfs, .. })`. The VFS resolves `navigator.storage.getDirectory()`, acquires the cross-tab writer lock (Web Locks), and opens `/main.db` via `createSyncAccessHandle()`.
3. All reads/writes/commits execute as direct synchronous handle calls into WASM linear memory — zero message round-trips, zero byte-array boxing. Futures returned by the `Vfs` trait resolve immediately.
4. Attempting to open on the main thread (or any non-dedicated-worker context) fails fast with a typed `WorkerOwnership`-class error — no hang, no panic (telpher maps this to `DocStoreError::BackendUnavailable`).

### Flow 2: Main-thread JS or off-worker Rust reaches the database (Mode 2, optional)

1. App opts into the async bridge at bootstrap. The storage worker still exclusively owns all sync access handles — and all `OpfsVfs` values: the bridge is strictly a message protocol; no Rust state crosses realms.
2. External contexts (main-thread JS, another worker, Rust async on a different thread) submit operations over the bridge as `postMessage` + transferable `ArrayBuffer`s (zero-copy ownership transfer, replacing the old `Array.from` byte-boxing protocol).
3. Correctness and v1 performance never depend on COOP/COEP. A SAB + Atomics transport for the same protocol is a capability-probed follow-up (§Follow-up Plans), not part of v1.

### Flow 3: Second tab opens the same database

1. Tab B's open attempt fails to acquire the Web Locks writer lock. Web Locks queue by default, so fail-fast is implemented with `{ ifAvailable: true }`: the request callback receives `null` on contention, which maps to `PagedbError::AlreadyLocked`; on success the callback holds a release-promise open until the Rust `LockHandle` drops (RAII → resolve promise → lock released).
2. The embedder surfaces the contention error. In v1 there is **no cross-tab reader path at the VFS level**: pagedb's ReadOnly/Observer open modes over OPFS are declared unsupported (typed error), because plain sync access handles are exclusive per file. Follower tabs that want reads proxy at the application level through the owning tab (telpher's leader/follower DocStore pattern) — the embedder's job, not the adapter's.
3. When Tab A closes or crashes, the browser releases its Web Lock; Tab B can acquire on retry. No configuration exists in which two tabs hold writable sync access handles to the same file. (Read-only-mode sync handles for true multi-tab readers are a compat-gated follow-up — see §Follow-up Plans.)

### Flow 4: Crash and reopen

1. The browser kills the storage worker mid-commit (tab close, OOM, process kill).
2. On next open, PageDB's existing recovery runs unchanged: A/B header seq selection tolerates a torn header, and open-flow **catalog reconciliation** recovers interrupted segment promote/tombstone renames (staged segments whose live copy is missing get promoted). Note: normal commits do not populate the apply-journal (`apply_journal_root_page_id: 0` in `WriteTxn::commit`) — the journal machinery exists but reconcile, not journal replay, is the recovery path the adapter's rename behavior leans on.
3. The durability oracle proves this in a real browser: forced worker termination at every commit phase — specifically including after header sync, before/after each segment rename, and during reconcile's own rename — then reopen must yield exactly the last committed state (all-or-nothing, receipts identical to the native oracle).

### Flow 5: Quota pressure and eviction

1. Bootstrap probe reports `navigator.storage.estimate()` and requests/reports `navigator.storage.persist()` status (Safari grants heuristically; arbitrary sites remain evictable).
2. A write that exhausts quota surfaces `QuotaExceededError` as a typed I/O error — never a WASM trap; the engine aborts the write and the transaction rolls back cleanly.
3. Because OPFS is evictable (Safari ITP eviction explicitly covers OPFS) and pagedb is pre-1.0, the adapter exposes the measurement hooks and a **manifest-aware logical-tree export/import**: export materializes the logical PageDB directory (real path names, not physical IDs) so the result opens byte-identically on native — raw physical-directory dumps are explicitly not the export format, since ID-named files are meaningless without the manifest (telpher-usage.md gate).

## Scope

### In Scope (v1)

- **In-worker sync `OpfsVfs` (Mode 1)** — full rewrite of `src/vfs/opfs/`: direct `FileSystemSyncAccessHandle` calls from Rust/wasm inside a dedicated worker; reads/writes pass `Uint8Array` views over WASM linear memory (no `Array.from`, no serde byte marshaling); vectored ops coalesced; `supports_direct_io() == false`.
- **Full `Vfs`/`VfsFile` contract** as defined in `src/vfs/traits.rs` and pinned by the `vfs_memory`/`vfs_tokio` conformance suites: `OpenMode` semantics, short reads + deterministic zero-fill past EOF on vectored reads, `truncate`/`set_len`/`len`, direct-child `list_dir`, idempotent `mkdir_all`/`remove`, fail-fast `lock_*` with RAII release, faithful `PagedbError` mapping (`Io(NotFound/AlreadyExists)`, `ReadOnly`, `AlreadyLocked`, `Unsupported`). "Full contract" is claimed only when the ported conformance suite — including `rename_while_open_keeps_handle_alive` — passes under real OPFS; any contract point OPFS genuinely cannot meet becomes an explicit, upstream-agreed carveout, not a silent divergence.
- **Vectored-write atomicity strategy** — the trait demands no-partial-state-observable vectored writes, but OPFS sync-handle writes are in-place and can fail mid-sequence (quota, I/O error). Pre-extending the file to its final length and pre-flighting `estimate()` are **advisory mitigations only** — `estimate()` is not a reservation and cannot guarantee quota stability across the subsequent writes. /spec must therefore either (a) implement a staging-and-swap path that delivers observable multi-write atomicity, or (b) obtain an explicit upstream carveout narrowing the contract to the commit protocol's actual tolerance of pre-header partial page writes, backed by a browser test demonstrating the accepted residual behavior. No third option.
- **Per-file handle registry in the worker** — dedupe/refcount so the engine's double-open of `/main.db` (cached Pager handle + per-commit `commit_header` reopen) works under OPFS exclusive locks; drop-close made deterministic (no fire-and-forget race with subsequent opens of spill files).
- **Rename via logical path indirection (primary design)** — physical OPFS files are named by opaque IDs; a persisted manifest maps VFS paths → physical IDs (the SQLite `opfs-sahpool` pattern). `rename(from, to)` is then a manifest update: open handles genuinely stay bound to the same data (trait satisfied, not simulated), rename needs no `move()` support, and the segment promote/tombstone crash window reduces to a manifest commit. Fallback if indirection is rejected upstream: close-handles → `move()`/copy → reopen-and-remap, with an explicit trait carveout and reconcile-based crash proof. The `rename_while_open_keeps_handle_alive` conformance test under real OPFS is the acceptance gate either way.
- **Manifest crash protocol (the indirection design's own durability story)** — the manifest is itself a tiny database and gets a specified mini-protocol, not an adjective: dual A/B manifest slots (own OPFS file(s)) carrying seq + integrity check (checksum or MAC), commit ordering = physical file data `flush()` → inactive manifest slot write → manifest `flush()`; open selects the highest-valid-seq slot; orphaned physical IDs (present on disk, absent from the winning manifest) are garbage-collected on open. `Vfs::sync_dir(path)` maps to a manifest flush for managed paths — giving `sync_dir` a real meaning on OPFS instead of pass 0's asserted no-op. **Upstream consequence**: several post-header metadata paths currently swallow side-effect errors — `sync_dir` in `txn/db/segment.rs:272` and `recovery/gc.rs`; `mkdir_all`/`rename` in `recovery/journal.rs` (`execute_journal_actions` returns `()`, and `txn/db/snapshot.rs` then proceeds to clear `apply_journal_root_page_id`, so a failed rename can be marked complete instead of retryable). Once rename/`sync_dir` carry manifest commits, a swallowed error is a swallowed manifest-commit failure. The /spec plan must audit **all post-header metadata side effects** (`mkdir_all`, `rename`, `remove`, `sync_dir`) across `txn/db/segment.rs`, `compaction/helpers.rs`, `recovery/reconcile.rs`, `recovery/gc.rs`, `txn/db/gc.rs`, `recovery/journal.rs`, `txn/db/snapshot.rs`, and `txn/write.rs` (the latter currently propagates errors — include it so the audit is complete, not because it is known-broken), and either propagate errors (e.g., `execute_journal_actions` returning `Result`) or explicitly preserve retryability before completion state is cleared. Exact manifest layout is /spec's job; required oracle cases are not: torn manifest write, manifest/data mismatch, crash after manifest commit but before orphan GC, rename issued during reconcile, and a swallowed-manifest-flush-failure probe.
- **Logical-tree semantics over the manifest** — with ID-named physical files, the OPFS directory is no longer the logical PageDB tree: `list_dir`, `remove`, `mkdir_all`, and open-flow reconcile operate on the **manifest-backed logical view**, never on physical directory walks. Debuggability: OPFS Explorer shows opaque IDs, so the harness ships a small manifest-dump/resolver dev helper.
- **Honest durability mapping** — `sync()` → `flush()` with per-browser semantics documented (ordering barrier + OS-level data sync; not power-fail-proof on macOS); `sync_dir()` maps to the manifest flush for managed paths (see manifest protocol above) rather than pass 0's asserted no-op; commit-protocol ordering (pages-sync → header-write → header-sync → renames → manifest/dir-sync) preserved.
- **Cross-tab single-writer** — Web Locks (`navigator.locks`) backing `lock_exclusive`/`lock_shared` on sentinel paths (`.writer.lock`, `.frozen_readers.lock`, `.observers.lock`). Because Web Locks queue by default, fail-fast is implemented with `{ ifAvailable: true }` (null callback → `AlreadyLocked`) and RAII release by holding the lock callback's promise open until `LockHandle` drop. Required tests: contention, drop-release, tab/worker termination release, and no accidental queued acquisition. **Multi-tab reader stance (v1)**: pagedb's ReadOnly/Observer open modes are unsupported over OPFS and must fail with a typed error. Mechanism is an explicit upstream decision, because `DbMode` is resolved above the VFS and **every option requires an upstream change**: `open_with_mode` currently erases `lock_shared` errors into `AlreadyLocked` (`txn/db/open.rs:552,559`), and ReadOnly probes `.writer.lock` first, returning `WriterPresent` while a writer is active (`open.rs:544`) — so a backend returning `Unsupported` from reader-sentinel `lock_shared` is masked today. Options for /spec to settle with the author: (a) preserve `Unsupported` through `open_with_mode`, or (b) a backend mode-capability check (a `Vfs` capability hook consulted inside `open_with_mode` **before** the writer-present probe, so the OPFS reader error is `Unsupported` rather than `WriterPresent`/`AlreadyLocked`). A JS bootstrap guard alone cannot satisfy the acceptance tests — direct `Db::open_read_only`/`open_observer` calls never pass through the bootstrap. Acceptance tests: `Db::open_read_only(OpfsVfs)` and `Db::open_observer(OpfsVfs)` return the intended typed error in **both** writer-present and writer-absent states. Follower tabs proxy reads at the application level through the owning tab.
- **Typed error taxonomy + storage surfacing** — Corruption (AEAD failure, already typed by pagedb), quota exhaustion, `WorkerOwnership` (main-thread open), `AlreadyLocked`, `Io`; every JS exception crossing wasm-bindgen mapped, never unwrapped. `PagedbError` already carries `Quota`, `NoSpace`, `AlreadyLocked`, and `Unsupported` (`src/errors.rs`) — the actual gaps are the **browser-storage mapping** (DOM `QuotaExceededError` → existing `Quota`/`NoSpace` or a new VFS-storage variant, upstream's call during /spec) and **worker-ownership** (likely a new or mapped variant). Quota-failure tests required at each write site: file create, page write, segment seal, header write, rename path. `estimate()`/`persist()` probe results exposed; instrumentation hooks for open/commit/commit-error/corruption with realm, bytes, duration (telpher observability contract §7).
- **Optional async bridge (Mode 2)** — worker-hosted VFS **service** callable from outside the worker over `postMessage` + transferable `ArrayBuffer`s (portable, replaces the current `Array.from` protocol). Strictly a message protocol: bridge clients never hold `OpfsVfs` or any shared Rust value — the `unsafe Send/Sync + send_wrapper` justification requires `OpfsVfs` to stay confined to its single worker realm. Off by default; correctness independent of COOP/COEP. (The SAB + Atomics transport fast path is deferred to §Follow-up Plans — same protocol, faster lane, and it would force a redesign of the Send/Sync justification if it ever shares Rust values.)
- **JS bootstrap + capability probe** (TS module in this repo): spawn storage worker, probe (OPFS? sync handles? `crossOriginIsolated`? persist status?), `openPersistent`-style init matching the shape telpher's docs already assume, typed error surface.
- **Browser test + bench harness (this repo, permanent)** — VFS conformance suite (port of `tests/vfs_memory.rs`/`vfs_tokio.rs`) plus pagedb engine suites (smoke, txn, btree, segment, crash, recovery, `page_size_range` — snapshot excluded: native-only `cfg`, stays upstream native regression coverage) running against real OPFS in headless Chromium + Firefox + WebKit; durability oracles (kill-worker-mid-commit at each phase, torn-header, reconcile + manifest oracle cases, logical export/import round-trip, two-tab contention torture); native↔web receipt-determinism oracle (BLAKE3 over committed state, per telpher DOCSTORE-PDB-007); benches (per-op VFS overhead vs raw sync handle, page read/commit throughput vs the current postMessage backend, page-size sweep 4–64 KiB).
- **Upstream PR to `NodeDB-Lab/pagedb`** — the rewritten backend + worker glue behind the existing `opfs` feature/type names, doc-drift fixes (README's nonexistent `gloo-worker` claim; `src/vfs/opfs/mod.rs` reference to a nonexistent `openPersistent` JS function), and a minimal upstream CI check; heavy browser CI stays here.

**Delivery order within v1** (Codex review finding — prove the storage contract before widening the surface): **M1** = Mode 1 backend + handle registry + rename indirection + conformance suite + crash oracle + upstream PR draft. **M2** = cross-tab Web Locks + quota/eviction surfacing + bootstrap + Mode 2 postMessage bridge + benches + PR finalization. Mode 2 does not gate M1 acceptance.

- **Mesh/P2P/distribution** (Zenoh, iroh, WebRTC/WebTransport, CRDTs, Headscale — the bulk of the three Gemini research docs) — separate project layered above the store.
- **Cross-tab reader/observer modes over OPFS** — pagedb's ReadOnly/Observer opens from a non-owning tab return typed errors in v1; read-only-mode sync handles (`{mode: 'read-only'}`, compat-gated) are a follow-up (§Follow-up Plans). App-level follower proxying through the owning tab is the embedder's responsibility.
- **SAB + Atomics bridge transport** — deferred to follow-ups; Mode 2 ships on `postMessage` + transferables only.
- **JS DB client over the bridge** (plain-JS open/get/put/txn API to the worker-hosted PageDB) — deferred; plan captured in §Follow-up Plans.
- **npm packaging with TS types** — deferred; plan captured in §Follow-up Plans.
- **Multi-writer** (cross-process or cross-tab concurrent writers) — pagedb non-goal; single writer + readers only.
- **IndexedDB fallback VFS** — a post-M1 candidate, not a shipped fallback. Firefox now proves the dedicated-worker binary transaction, raw transaction-abort atomicity, and an experimental local-only `IdbVfs` with shared file-handle state, rename while open, separate `sync`/`sync_dir` boundaries, VFS-level file and namespace transaction abort recovery, post-commit orphan cleanup with retry after an aborted sweep, real worker termination immediately before, during, and after namespace publication plus after a header write before its persistence sync, a native-matching full-reopen receipt script, and cross-worker writer contention plus post-termination Web Locks release. It still needs Chromium capability proof, cross-tab coverage, quota/error injection, request-failure recovery, and the full conformance plus receipt-oracle matrix. Until those gates pass, browsers without OPFS sync handles return a typed `BackendUnavailable`-class error.
- **COOP/COEP as a correctness requirement** — cross-origin isolation is an optional performance upgrade only (both consumers mandate this).
- **`Vfs` trait redesign** — the adapter satisfies the existing trait; any trait-level change (e.g., relaxing `Send` bounds on wasm) needs upstream agreement and is a deviation to raise during /spec, not assumed here.

## Technical Context

*Lightweight notes for /spec — the full contract lives in the pagedb source and the conversation records.*

- **The contract**: two RPITIT async traits in `pagedb/src/vfs/traits.rs` — `Vfs` (`open/remove/rename/list_dir/mkdir_all/sync_dir/lock_exclusive/lock_shared`, assoc. `File: VfsFile`, `LockHandle`; impls must be `Send + Sync + Clone`) and `VfsFile` (`read_at/read_at_vectored/write_at/write_at_vectored/sync/truncate/set_len/len/is_empty/supports_direct_io`, futures `Send`). `MemVfs` (`src/vfs/memory.rs`) is the documented reference semantic; `tests/vfs_memory.rs` + `tests/vfs_tokio.rs` pin observable behavior. On wasm32 the existing pattern is `unsafe impl Send/Sync` guarded by `send_wrapper`, whose soundness argument assumes exactly one JS thread per `OpfsVfs` — therefore `OpfsVfs` is confined to its single worker realm, all external access goes through the Mode 2 message protocol (never shared Rust values), and any future shared-memory WASM work must redesign that justification first.
- **Commit protocol the VFS must not break**: CoW shadow paging, A/B header slots at pages 0/1 of `/main.db` (seq-numbered, MAC-verified, torn-write-tolerant), ordering = dirty pages vectored-write + `sync()` → inactive header write + `sync()` → segment renames (staging→live promote, live→tombstone) + `sync_dir()`. Page sizes: 4096–65536 (5 steps). Recovery: header seq selection + open-flow catalog reconcile (the operative path for interrupted renames — normal commits write `apply_journal_root_page_id: 0`, so the pages-2/3 journal machinery is not exercised by them) + observer AEAD-retry.
- **Paths**: VFS-root-relative (`/main.db`, `seg/<hex32>`, `seg/.staging/…`, `seg/.tombstone/<hex>.<commit>`, `tmp/scratch-N`, `.writer.lock` …). Engine holds multiple live handles to `/main.db` simultaneously — the worker-side handle registry is not optional.
- **Existing backend to replace**: `src/vfs/opfs/{mod.rs,protocol.rs,vfs_impl.rs,handle.rs,opfs_worker.js}` behind `feature = "opfs"`; public API today = `OpfsVfs` + `OPFS_WORKER_JS`. Keep names/feature stable for consumers unless upstream prefers otherwise.
- **Platform facts (researched, primary sources)**: see §Research Findings. Load-bearing: sync handles are DedicatedWorker-only with exclusive per-file locks (`NoModificationAllowedError` on double-open); the `mode` option is a non-standard Chromium extension (Chrome ~121+); `move()` is unstandardized, takes locks on source+destination (no rename-while-open), atomicity not guaranteed, Firefox diverges; `flush()` is spec-level best-effort — Chromium implements `fdatasync` (Linux/Android) / `F_BARRIERFSYNC` (macOS/iOS, ordering-not-power-fail); `close()` does not flush; Safari eviction covers OPFS, `persist()` heuristic-granted.
- **Consumers**: telpher `document-editing-stack/docstore-pagedb.md` (DocStore trait, `OpenOptions { vfs }`, oracle tests DOCSTORE-PDB-001–007, observability §7, worker-ownership §8.6) and gitspen `PRD-modular-stack.md` §5 + `crates/nodedb-lite` (wasm-clean dependency set — no full tokio runtime), `crates/storage-pagedb` (vendored pagedb via git/path; pagedb is not on crates.io). Both repos: jj-colocated, mise-pinned, justfile; this repo should adopt the same meta kit. License: MIT OR Apache-2.0 to match pagedb.
- **MVCC semantics across targets**: pagedb's snapshot isolation (CoW shadow paging — readers pin old roots while the writer CoW-allocates and publishes via the A/B header swap; the freelist respects reader pins) lives entirely *above* the VFS, so Mode 1 preserves it unchanged: the adapter's commit-ordering contract **is** the MVCC-preservation contract, and long-lived read transactions interleaving with commits on the worker's async executor get native-identical snapshot guarantees. What the browser constrains: (1) cross-*process* reader MVCC (native ReadOnly/Observer processes) maps to cross-*tab* readers — unsupported in v1 (typed error); Follow-up #4's read-only-mode handles restore it, and observer mode's header-seq selection + AEAD-failure retry already tolerate a concurrent writer, so the engine machinery transfers mechanically; (2) reader/writer concurrency in v1 is logical (interleaved futures on one worker thread), not physically parallel — parallel readers need multi-handle modes or shared-memory WASM threads, both fenced behind follow-ups. Mode 2 callers reach the same single-worker `Db`, so they get in-process MVCC semantics over the message hop.
- **Async runtime caveat**: pagedb only returns futures the embedder drives; it uses no executor internally, but `tokio::time::sleep` appears in the observer retry loop (`src/pager/core.rs:516`) and tokio's wasm32 feature set is limited — whether that future resolves under `wasm-bindgen-futures` without a tokio reactor is unverified (§Verify-during-/spec). In-worker sync calls also block the worker thread during I/O; acceptable for a storage worker, but the /spec must confirm nothing in `Db` requires interleaved future progress during a commit.

## Key Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Core architecture | DB runs inside one dedicated storage worker; VFS calls sync handles directly ("Mode 1") | Fastest possible path (zero message hops, zero copies); matches the proven SQLite `opfs-sahpool` shape; satisfies both consumers' worker-ownership mandate; needs no COOP/COEP |
| Cross-thread access | Optional bridge (Mode 2): postMessage+transferables, message-protocol only; SAB transport deferred to follow-ups | User wants external JS/Rust-async callers as an option, not a default; consumers forbid SAB as a correctness dependency; transferables fix the current backend's `Array.from` disaster; message-only keeps the wasm `Send/Sync` justification sound |
| Browser floor | Portable baseline: Chrome/Edge/Firefox/Safari on plain sync handles; Chromium extras (`readwrite-unsafe`, `move()`) detected fast paths only | `mode` option and `move()` are unstandardized Chromium extensions; gitspen plans Safari support; telpher forbids header-gated correctness |
| Double-open handling | Worker-side per-file handle registry (dedupe/refcount), not `readwrite-unsafe` | Registry is portable everywhere; `readwrite-unsafe` is Chromium-only and removes browser-level protection |
| Rename | Logical path indirection (ID-named physical files + crash-safe manifest, sahpool-style); close-move-reopen only as upstream-rejected fallback with explicit carveout | OPFS cannot move a file with an open sync handle (spec locking model); indirection satisfies rename-while-open *genuinely* instead of simulating it, and reduces the promote/tombstone crash window to atomic manifest replacement |
| Multi-tab readers | v1: ReadOnly/Observer opens from non-owning tabs are typed errors; app-level proxying via owning tab; read-only-mode handles compat-gated follow-up | Plain sync handles are exclusive per file; pretending pagedb's reader modes work over OPFS would corrupt or deadlock; consumers' leader/follower designs already proxy at app level |
| Durability claims | Document per-browser `flush()` reality (ordering barrier / OS data sync; not power-fail-proof on macOS); rely on pagedb's ordering-based commit protocol | Spec makes `flush()` best-effort; Chromium consciously chose `F_BARRIERFSYNC` citing SQLite precedent; pagedb's A/B protocol needs ordering more than absolute durability — honest claims beat false fsync guarantees |
| Cross-tab safety | Web Locks with `{ifAvailable: true}` + held-open callback promise for RAII; second tab fails fast with `AlreadyLocked` | Web Locks queue by default — `ifAvailable` is the only fail-fast mode, and RAII release requires keeping the lock callback's promise pending until `LockHandle` drop; browser releases locks on tab death; consumers specify exactly this pattern |
| Delivery split | Backend PR'd upstream to `NodeDB-Lab/pagedb` (author PR-friendly); harness/benches/demo permanent in `pagedb-opfs` | Keeps "PageDB handles OPFS natively" true for consumers; upstream stays lean; fast local iteration loop |
| JS surface v1 | Worker bootstrap + capability probe only | The piece telpher's docs already assume; JS DB client + npm packaging deferred to follow-ups |
| Testing bar | Real-browser conformance + durability oracles + receipt parity are v1 acceptance, not roadmap | The current backend has never executed; telpher hard-gates adoption on the oracle; "full reliability" is unprovable without it |
| Research handling | Deep-research primary-source extractions embedded with citations; unresolved items explicitly listed for /spec | Verification pass truncated by spend limit; claims carry sources; nothing unverified is load-bearing without a /spec check |

## Research Findings

Primary-source extractions (adversarial verification incomplete — treat each as sourced-but-single-pass):

**Locking & modes**
- Spec: `createSyncAccessHandle()` takes an exclusive lock on the file entry; further sync handles or writable streams are blocked until `close()`; failed acquisition rejects with `NoModificationAllowedError`. Sync handles are IDL-restricted to dedicated workers and the bucket file system. — [fs.spec.whatwg.org](https://fs.spec.whatwg.org/#dom-filesystemsyncaccesshandle-flush)
- The `mode` option (`'readwrite'` default, `'read-only'`, `'readwrite-unsafe'`) shipped in Chrome ~121 and is absent from the WHATWG standard (zero spec matches) — a Chromium/WICG extension. — [developer.chrome.com](https://developer.chrome.com/blog/new-dev-trial-for-multiple-readers-and-writers)

**Rename / move()**
- `FileSystemHandle.move()` remains unstandardized: whatwg/fs PR #10 open and unmerged since 2022 (confirmed via GitHub API, 2-0 verification vote); browsers ship divergent forms. Spec direction: POSIX-style silent overwrite; `move()` takes locks on **both** source and destination → no move while a sync handle is open; atomicity explicitly not guaranteed; Firefox blocks moves under any held lock and historically threw on overwrite. — [github.com/whatwg/fs#10](https://github.com/whatwg/fs/pull/10)

**Durability**
- Spec defines `flush()` as an *attempt* to persist, permitted to be a no-op on some file systems; `close()` does **not** guarantee persistence (and no browser flushed on close as of Jan 2023 — the spec was rewritten to match reality). The flush-durability question (whatwg/fs #156, #71) remains open with no normative resolution. — [fs.spec.whatwg.org](https://fs.spec.whatwg.org/#dom-filesystemsyncaccesshandle-flush), [whatwg/fs#156](https://github.com/whatwg/fs/issues/156), [whatwg/fs#71](https://github.com/whatwg/fs/issues/71)
- Chromium `base::File::Flush()`: `fdatasync()` on Linux/Android/ChromeOS/Fuchsia; `F_BARRIERFSYNC` on macOS/iOS — deliberately downgraded from `F_FULLFSYNC` for latency after a corruption experiment, citing SQLite's `fullfsync=off` default. Write-ordering barrier, not full power-loss durability. WebKit avoided `F_FULLFSYNC` long ago. — [chromium file_posix.cc](https://github.com/chromium/chromium/blob/main/base/files/file_posix.cc), [whatwg/fs#156](https://github.com/whatwg/fs/issues/156)

**Quota & eviction (Safari/WebKit)**
- Safari 17+: per-origin quota up to ~60% of disk (browser apps) / 15% (other apps); `QuotaExceededError` on exhaustion; no user prompt. WebKit eviction under ITP explicitly covers File System (OPFS) data. `persist()` granted heuristically (Home-Screen/Dock installs); arbitrary origins stay best-effort evictable. — [webkit.org storage policy](https://webkit.org/blog/14403/updates-to-storage-policy/)

## Success Criteria

1. **Full function**: the ported VFS conformance suite (mirror of `tests/vfs_memory.rs`/`vfs_tokio.rs` — explicitly including `rename_while_open_keeps_handle_alive` and the vectored all-or-nothing/zero-fill tests, with any upstream-agreed carveouts documented in the same PR) and pagedb's engine-level suites (smoke, txn, btree, segment, crash, recovery, page-size range) pass against real OPFS in headless Chromium, Firefox, and WebKit. The snapshot suite is excluded from the browser matrix — `src/snapshot/` is `cfg`-gated native-only (it uses `tokio::fs`); its browser counterpart is the manifest-aware logical export/import oracle in criterion 2.
2. **Crash consistency (browser-observed)**: durability oracle green — worker killed at every commit phase (including after header sync, around each segment rename, and during reconcile's rename) reopens to exactly the last committed state; torn-header and reconcile cases recover; manifest oracle cases pass (torn manifest write, manifest/data mismatch, crash between manifest commit and orphan GC, rename during reconcile); two-tab torture test produces typed contention errors and zero corruption; native↔web receipt determinism (BLAKE3 state parity) holds, including the **export round-trip**: logical-tree export from OPFS opens on native with identical receipts. *Scope of the claim*: logical atomicity across worker/tab termination under test — **not** power-loss durability, which `flush()` cannot promise cross-browser (§Research Findings).
3. **Max performance**: benches show Mode 1 per-op overhead within a small constant factor of raw in-worker sync-handle calls (target set at /spec from prior-art baselines), and ≥ an order of magnitude improvement over the current postMessage backend on 4 KiB page reads; no `Array.from`/byte-boxing anywhere on the hot path.
4. **Typed failure surface**: main-thread open, double-tab open, quota exhaustion at each write site (create, page write, segment seal, header write, rename path), and corruption each produce their designated typed error in tests — no traps, no hangs, no partial-commit visibility after quota failure.
5. **Upstream landed or PR-ready**: backend + doc-drift fixes packaged as a reviewable PR against `NodeDB-Lab/pagedb` with upstream CI check.

2. **npm packaging** — publish the bootstrap (and client) as a typed npm package with worker asset handling for common bundlers (vite/webpack/bun). Depends on: bootstrap API frozen, license/provenance notes (pagedb vendored via git/path, MIT OR Apache-2.0).

Depends on: Mode 2 protocol frozen; a reviewed redesign of the wasm `Send/Sync` soundness argument if anything beyond raw bytes is shared.
4. **Read-only multi-tab readers** — `createSyncAccessHandle({mode: 'read-only'})`-based ReadOnly/Observer support from non-owning tabs, compat-gated per browser (the `mode` option is a non-standard Chromium extension today). Depends on: cross-browser support matrix; pagedb reader-mode semantics validated against invalidation-free OPFS reads.
