# iPhone Chrome Simulator Runbook

This runbook covers the no-attached-phone routes for exercising the Chrome for
iOS app shell. It does not replace the iPhone Safari durability proof: Chrome
for iOS uses the iOS WebKit engine class, so the separate Chrome target proves
app-shell behavior rather than an independent storage engine.

## Artifact Boundaries

There are three materially different artifacts:

1. **Google Chrome from the App Store** is a device application. Its IPA is not
   a durable Apple Simulator fixture.
2. **Google CI `Chromium.app`** is generated by `chrome/ci/ios-simulator` and
   archived as `gs://bling-archive/<version>/<timestamp>/Chromium.tar.gz`.
   Chromium's own tooling says this is similar to, but not actually, a release
   build. It is the best prebuilt local artifact when an authorized Google
   identity can read the bucket.
3. **Source-built `Chromium.app`** is the public, reproducible local fallback.
   It comes from Chromium's public iOS source and omits proprietary Google
   Chrome branding and services.

The official Chrome for Testing download service does not currently publish an
iOS or iOS Simulator platform. Do not substitute an unsigned third-party app
bundle whose source revision, builder, and update policy cannot be verified.

## Trustworthy Prebuilt Audit

The final 2026-07-15 audit did not find a trustworthy, publicly downloadable
iOS Simulator app:

- Chrome for Testing publishes Linux, macOS, and Windows downloads, but no iOS
  platform.
- The public `chromium-browser-snapshots` bucket has no iOS snapshot prefix.
- Google's `bling-archive`, `chrome-unsigned/ios-G1N`, and
  `chrome-signed/ios-G1N` locations all require private bucket IAM. The latter
  two are release/device-IPA locations, not a public Simulator distribution.
- The public `chromium/ci/ios-simulator` LUCI builder exposes successful build
  status and source revisions. Anonymous build metadata does not expose a
  distributable `Chromium.app`; the builder is a revision oracle, not a binary
  release service.
- BrowserStack and Sauce Labs provide credentialed remote-device services,
  not a locally retainable Simulator app with public provenance.

The source builder therefore uses the newest successful public LUCI revision
as its default source pin and builds the documented `chrome` target locally.

## Remote OAuth From A Pixel

The Google Cloud CLI installed on this host must use its Homebrew Python 3.11;
macOS `/usr/bin/python3` is Python 3.9 and is unsupported by the installed CLI.

Set the identity in a visible Mac terminal controlled from the Pixel:

```sh
export CLOUDSDK_PYTHON=/opt/homebrew/opt/python@3.11/bin/python3.11
export ACCOUNT_EMAIL='your-google-account@example.com'

gcloud auth login "$ACCOUNT_EMAIL" \
  --brief \
  --no-activate \
  --no-launch-browser
```

Open the printed Google URL on the Pixel, complete consent for the named
identity, and paste the resulting authorization code directly into the visible
Mac terminal. The authorization code is credential material: never paste it
into chat, a Markdown file, shell history, an issue, or a test report.

`--no-activate` preserves the currently active Google Cloud identity. Every
archive command below selects the intended identity explicitly with
`--account="$ACCOUNT_EMAIL"`.

Confirm only that the requested identity was stored:

```sh
gcloud auth list \
  --filter="account=$ACCOUNT_EMAIL" \
  --format='table(account,status)'
```

## Probe The Google CI Archive

Resolve the current stable Chrome version from Google's public version index:

```sh
VERSION="$(
  curl -fsSL \
    https://googlechromelabs.github.io/chrome-for-testing/last-known-good-versions.json |
  jq -r '.channels.Stable.version'
)"
```

Probe the exact archive pattern without downloading anything:

```sh
PATTERN="gs://bling-archive/${VERSION}/*/Chromium.tar.gz"
gcloud storage ls "$PATTERN" --account="$ACCOUNT_EMAIL"
```

Interpret the result strictly:

- One or more object paths: the identity has read access and that version is
  archived.
- `matched no objects`: access may work, but that exact version is absent; try
  the Beta, Dev, or Canary version from the same JSON document.
- `403 AccessDenied`: the identity does not have the required bucket IAM.
- Authentication or token errors: repeat OAuth for the same identity; do not
  treat these as an IAM result.

There is no legitimate command-line bypass for a `403`. A Google-managed
identity with access must grant the bucket permission, or the source-build
fallback must be used.

## Download And Preserve Evidence

After a successful listing, select the newest timestamped object and preserve
its source path and metadata alongside the download:

```sh
ARCHIVE="$(
  gcloud storage ls "$PATTERN" --account="$ACCOUNT_EMAIL" |
  sort |
  tail -1
)"
DEST="$HOME/cache/rust-browser-proofs/google-ios-chromium/$VERSION"

mkdir -p "$DEST"
printf '%s\n' "$ARCHIVE" >"$DEST/gcs-object.txt"
gcloud storage ls --long "$ARCHIVE" --account="$ACCOUNT_EMAIL" \
  >"$DEST/gcs-object-metadata.txt"
gcloud storage cp "$ARCHIVE" "$DEST/Chromium.tar.gz" \
  --account="$ACCOUNT_EMAIL"
shasum -a 256 "$DEST/Chromium.tar.gz" \
  >"$DEST/Chromium.tar.gz.sha256"
tar -xzf "$DEST/Chromium.tar.gz" -C "$DEST"
```

Validate that the archive contains the expected Simulator application and
record its bundle identity:

```sh
test -f "$DEST/Chromium.app/Info.plist"
plutil -extract CFBundleIdentifier raw \
  "$DEST/Chromium.app/Info.plist" |
  tee "$DEST/bundle-id.txt"
```

The locally calculated SHA-256 detects later cache corruption. It is not an
independent Google signature; provenance still rests on authenticated download
from Google's bucket and the recorded object path.

The repository runner boots the configured simulator, installs the app when
needed, derives its bundle ID, and launches the requested URL:

```sh
cd /Users/andrew/code/pres/brow/rust-browser-proofs

IPHONE_CHROME_APP_PATH="$DEST/Chromium.app" \
  just run-iphone-chrome 'https://example.com/'
```

Then verify discovery independently:

```sh
just check-iphone-chrome
```

Passing this gate proves the iOS Chromium app shell can be installed and
launched. It does not convert the result into a branded Google Chrome release
proof and does not supersede the iPhone Safari WebKit durability matrix.

## Source-Build Fallback

When neither identity has archive IAM, use the repository-owned source builder:

```sh
cd /Users/andrew/code/pres/brow/rust-browser-proofs
just build-iphone-chromium-source
just run-iphone-chromium-source 'https://example.com/'
```

The default root is `~/.volumes/chromium`. The script rejects roots under
`~/code`, requires a configurable 150 GiB free-space guard before the initial
checkout, and keeps this layout:

```text
~/.volumes/chromium/
  depot_tools/     # pinned by recorded Git revision
  git-cache/       # fetch --git-cache storage
  checkout/src/    # Chromium checkout and out/Debug-iphonesimulator
  logs/            # complete source-build logs
  provenance/      # LUCI response and selected source revision
  artifacts/
    <revision>/    # retained Chromium.app, GN args, metadata, file hashes
    current -> <revision>
```

The first run selects the newest successful public iOS builder revision and
persists it in `provenance/source-revision.txt`. Retries reuse that exact pin.
Set `CHROMIUM_REVISION` to a 40-character commit for an explicit override, or
set `CHROMIUM_REFRESH_REVISION=1` to intentionally query LUCI for a newer
successful revision. Set `CHROMIUM_BUILD_JOBS` only when an explicit Ninja
concurrency limit is needed. Set
`CHROMIUM_HEARTBEAT_SECONDS` to change the terminal heartbeat interval from its
30-second default. Set `CHROMIUM_IOS_ROOT` only to another large-volume path
outside `~/code`.

Verbose `fetch`, `gclient`, GN, and Ninja output is written to the timestamped
file reported as `log_path`. The terminal receives bounded phase transitions
and heartbeats; on failure it receives only a capped, carriage-return-sanitized
tail while the complete diagnostic record stays in the log. This prevents
Chromium's progress renderers from flooding or freezing a terminal multiplexer.

The macOS compile runs under a filesystem sandbox that denies reads from
`~/node_modules`. Chromium's pinned source and its hash-verified official
dependency archives remain readable under `~/.volumes/chromium`; unrelated
host JavaScript packages cannot satisfy or alter transitive TypeScript module
resolution. The denied path is recorded in `build-provenance.txt`.

Chromium documents roughly 30 GiB for the shared Git cache and warns that the
initial fetch can take from 30 minutes to several hours. The complete checkout
and build output require additional space. Re-running the recipe reuses the
existing depot_tools checkout, Git cache, source checkout, and Ninja outputs.

The source runner sets Chromium's test-only `FirstRunForceDisabled` preference,
disables `BuildExternalPrivacyContext` for unbranded Chromium, and terminates
the prior process before delivering one URL. This is required because a server
variation can enable a provider path that the public identity stub implements
as fatal `NOTREACHED()`. The runner waits 15 seconds after the handoff, requires
the app process to remain present, and fails if a new Chromium crash report
appears. These guards are limited to the source-built test app and are not
applied to branded Google Chrome.

## Exact Branded Chrome Proof

If the acceptance criterion is the exact App Store-branded Google Chrome app,
the local Simulator routes above are insufficient. Use either:

- a Google-provided internal Simulator bundle, when legitimately available; or
- a reputable real-device cloud such as BrowserStack, explicitly tracked as a
  credentialed external test lane.

Do not label source-built Chromium or the Google CI `Chromium.app` as branded
Chrome in reports.

## Credential Cleanup

After testing, credentials can be removed without changing other identities:

```sh
gcloud auth revoke "$ACCOUNT_EMAIL"
```

Keep credentials only when this host is trusted and recurring archive access is
intentional.
